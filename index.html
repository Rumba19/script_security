Tab 1: XSS via innerHTML (vulnerable). Tab 2: runtime eval/new Function (vulnerable). Tab 3: mitigations (safe).

About this demo: The input below is inserted into the page using innerHTML without validation or encoding. Any HTML provided by the user is interpreted by the browser, so scripts or event handlers will execute. This is a classic reflected XSS pattern.

  • Vulnerable sink: element.innerHTML = userInput;
  • Risk: attackers can run arbitrary JS in the origin, steal cookies, or deface the UI.
  • How to test (form): paste <img src=x onerror=alert(1)> and submit.
  • How to test (URL): append ?q=%3Cimg%20src%3Dx%20onerror%3Dalert(1)%3E to the page URL.
  • Safer alternative: render untrusted data with textContent or sanitize with DOMPurify.
This tab is intentionally insecure for demonstration. Do not deploy this pattern.
Also via query: ?q=<img src=x onerror=alert(1)>

About this demo: These controls execute user-supplied strings as JavaScript using eval() and new Function(). This enables remote code execution in the page context. Any code the user enters runs with the page’s privileges.

  • Vulnerable sinks: const r = eval(expr); and const fn = new Function(body); fn();
  • Risks: data theft, CSRF via scripted actions, DOM defacement, supply-chain pivots.
  • How to test (eval): enter alert("pwned") or 2+2 then run.
  • How to test (Function): enter return 7*6 or alert("pwned") then run.
  • Why unsafe: parses and executes arbitrary code at runtime; CSP often cannot fully mitigate.
  • Safer alternatives: parse data (e.g., JSON.parse), whitelist operations in code, use explicit APIs, avoid string-based timers/handlers (setTimeout("...")), and never accept code as input.
This tab is intentionally insecure. Do not use eval or new Function with untrusted input. 

        


      
      

        

        

            About these mitigations: Untrusted input is rendered as text with
            textContent. When minimal HTML is required, input is sanitized via DOMPurify
            using a strict allowlist. No dynamic code execution is performed.
        

        

        
    
            
  • Primary defense: element.textContent = userInput; encodes characters so the browser does not interpret them as HTML.
    • 
            
  • When HTML is needed: sanitize first:
            DOMPurify.sanitize(userInput, { ALLOWED_TAGS:[ "b","i","u","strong","em","br","a" ], ALLOWED_ATTR:[ "href","title" ] }).
            
    • 
            
  • Link hardening: set target="_blank" + rel="noopener noreferrer" on sanitized anchors.
    • 
            
  • Avoid dynamic eval: never use eval, new Function, or string-based timers/handlers.
    • 
            
  • Extra hardening (server): apply a Content Security Policy (CSP) that disallows inline scripts and eval.
    • 
            
  • Developer guardrails: enable ESLint rules no-eval, no-implied-eval, no-new-func.
    •
